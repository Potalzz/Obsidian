
## 1과목 소프트웨어 설계
### 재공학
기존거 다시 씀.

장점: 비용, 위험 감소
목표: 유지보수 향상
과정: 분석-> 구성(구조 변경)-> **역공학** -> 이식

**역공학**
소프트웨어를 분석하는 기법
기존의 아이를 가지고 문서화를 통해 쓸만한 곳에 쓸 수 있느냐

### CASE(Computre Aided Software Engineering)
소프트웨어 엔지니어링을 도와주는 자동화 도구

**제공하는 기능**
- 개발 신속 정확하게
- 생명주기 전체 단계 연결
- 문서화 및 명세화를 위한 그래픽 기능 제공
- 개발 단계의 표준화를 기할 수 있으며, 자료 흐름도 작성 기능 제공
- 모델들 사이 모순 검사 기능 제공하고 다양한 소프트웨어 개발 모형 지원
- **원천기술**
	- 구조적 기법
	- 프로토타이핑 기술
	- 정보 저장소 기술


**CASE 사용의 장점**
- 기간 줄고, 비용 절약돼서 생산성 향상
**결론적으로 문서화 도구이다**

**CASE 분류**
상위: 요구분석 및 설계
하위: 실제 구현
통합: 소프트웨어 개발 주기 전체 과정 지원

**요구사항 분석을 위한 CASE 도구**
**SADT**
- SoftTech에서 개발, 블록 다이어그램 지원


### 소프트웨어 개발 방법론

**소프트웨어 생명 주기**
타당성 검토 -> 개발 계획 -> 요구사항 분석 -> 설계 -> 구현 -> 테스트 -> 운용 -> 유지보수

고전적인 모델
물이 떨어지듯 선형 순차적 모델. 소규모 개발에 적합
중간에 이벤트가 발생하면 적용이 어려움.

**하향식과 상향식 설계**
하향식: 가장 중요한 애를 만들고 나머지를 만들어나간다 (가지치기)
상향식: 기본적인 애들을 모아서 기둥을 세우는 형태

**프로토타입 모형**

**HIPO(Hierarchy Input Process Output)**
(계층적 입력 처리 출력)

입력, 처리, 출력으로 구성되는 시스템 분석 및 설계와 시스템 문서화 기법.
(폭포수형과 비슷)

가시적 도표 덕에 이해하기 쉬우며 유지보수가 쉽다.
하향식 소프트웨어 개발을 위한 문서화 도구.

### 애자일 방법론
**애자일 선언문**
고객과 의사소통해서 고객이 원하는거 만들어주자

#### XP(eXtremeProgramming)
아주 빠르게 양질의 소프트웨어를 만듬

**XP 핵심 가치**
- 소통
- 단순성
- 피드백
- 용기(고객 변화에 능동적으로 대응)
- 존중(팀원 간 상호 존중)

**XP 절차(Process)**
![[Pasted image 20250504164907.png]]
핵심은 고객이 원하는 것이 무엇인지 계속해서 반복하는 것이다.

Spike: 어려운 요구사항, 잠재적 솔루션을 고려하기 위해 작성하는 간단한 프로그램
User Stories: 사용자의 요구사항을 간단한 시나리오로 표현

*User Story:* 일종의 요구사항. 형식 X, 고객에 의해 작성
*Release(배포) Planning:* 몇 개의 스토리가 적용되어 부분적으로 기능이 완료된 제품을 제공하는 것으로, 완료 시점에 대한 일정 수립
*Iteration:* 하나의 릴리즈를 세분화한 단위이며, 1~3주 단위로 진행. 반복 진행 중 새로운 스토리가 추가 될 때 진행 중 반복이나 다음 반복에 추가될 수 있다.
*Acceptance Test:* 적응 테스트
*Small Release:* 작은 단위 별로, 실제 프로그램을 실행할 수 있는 단위로 만드는 것.

**XP의 12가지 실천사항(Practice)**

>**Fine Scale Feedback**
- Pair Programming(짝 프로그래밍)
	- 한 사람은 코딩, 다른 사람은 검사를 수행
- Planning Game
	- 게임처럼 선수와 규칙, 목표를 두고 기획
- Test Driven Development(TDD)
	- 코드 작성 전 단위 테스트부터 작성
- Whole Team
	- 사용자(고객)까지 팀에 속함

> **Continuous Process**
-  Continuous Integration
	- 상시 빌드 및 배포를 유지할 수 있는 상태로 유지
- Design Improvement
	- 기능 변경 없이 중복, 복잡성 제거, 소통 향상, 단순화, 유언성 등을 위한 재구성 수행
- Small Releases
	- 짧은 주기로 잦은 릴리즈 함으로써 고객이 변경사항 볼 수 있게

> **Shared Understanding**
- Coding Standards
	- 표준 코딩 기법에 맞춰서 작성
- Collective Code Ownership
	- 소스코드는 모든 프로그래머가 주인이다
- Simple Design
	- 최대한 간결하게
- System Metaphor
	- 최종적으로 개발되어야 할 시스템 구조 기술

> **Programmer Welfare**
- Sustainable Pace
	- 개발자에게 너무 과도한 업무를 주지 않는다

**효과적인 프로젝트 관리를 위한 3대 요소**
> **3P**
> 사람(People) - 인적 자원
> 문제(Problem) - 문제 인식
> 프로세스(Process) - 작업 계획

**SPRINT(전력질주)** 를 2주~4주단위로 반복하며, 데일리 스크럼 미팅을 통해서 24시간에 한 번씩 진행 사항과 만료된 것을 확인한다. 매일 소멸차트를 통해 작업한 일을 소거해나감.

##### SCRUM팀의 역할
**제품 책임자**
- 개발 목표에 이해도가 높은 개발 의뢰자(담당자)
- 제품의 요구사항을 파악하여 기능 목록 작성
- 제품 테스트 수행 및 요구사항 우선순위 갱신
- 업무 관점에서 우선순위와 중요도 표시
- 스프린트 계획 수립까지만 입무 수행
- 스프린트 시작되면 팀 운영에 관여 X

**스크럼 마스터**
- 업무를 배분만 진행. 강요 X
- 개발 과정에서 장애 요소 찾아서 제거

**스크럼 팀**
- 나머지 팀원들(개발자, 디자이너, 제품 검사기 등 모든 팀원)이 해당하고 팀원은 5~9명 내외로 구성
- 기능을 작업 단위로 분류, 요구사항을 사용자 스토리로 도출, 구현
- 일정, 속도를 추정한 뒤 제품 책임자에게 전달
- 스프린트 결과물을 제품 책임자에게 시연
- 매일 스크럼 회의에 참여하여 진행 상황 점검

**스크럼의 핵심은 아침에 일어나는 미팅(Daily Scrum Meeting)이고, 짧은 시간(약 15분)동안 서서 모든 회의를 끝냄.**

### 현행 시스템 분석
**소프트웨어를 개발 하기 위해 이 회사가 어떻게 굴러가는지 확인하는 것**

**목적**: 개발 시스템의 개발 범위를 확인, 이행 방향성 설정

**현행 시스템 파악 절차**
*1단계*
시스템 구성 파악 - 시스템 기능 파악 - 시스템 인터페이스 현황 파악
- 시스템이란 회사의 조직을 의미. 업무 흐름이 어떻게 되는지, 부서의 구성과 기능 파악.

*2단계*
아키텍처 파악 - 소프트웨어 구성 파악
- 아키텍처 = 설계도. 회사의 조직도와 운영되고 있는 소프트웨어 구성 파악

*3단계*
시스템 하드웨어 현황 파악 - 네트워크 구성 파악
- 사용하는 기기와 네트워크 파악

**시스템 아키텍처**
컴퓨터 시스템이 아닌, 조직의 구조와 어떻게 움직이는지 짜 놓은 것. 즉, 조직도다.
