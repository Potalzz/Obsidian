#### 의존성 주입(Dependency Injection)
상위 -> 하위가 아니라 하위 -> 상위로 의존해야 하는 원칙.
필요한 것을 외부에서 주입받는다.

**인젝션**
외부에서 생성된 객체를 전달받아 사용하는 구조

**프로토콜(인터페이스)**
프로토콜은 일종의 규격이다.
리모콘과 배터리를 예로 들어보자면 해당 규격만 준수하면 어떤 배터리든 장착할 수 있다.

**의존성 주입의 장점**
- 유지보수: 객체 간 결합도를 낮춰 유연한 대처 가능
- 테스팅: Mock데이터를 주입하여 테스트 가능
- 재사용성: 다양한 구현체를 손쉽게 교체 가능.

의존성 주입에는 다양한 방법이 있지만, 주로 생성자 주입(init)을 보편적으로 사용한다.

**추상화 & 주입을 통해 유지보수 가능한 구조를 만드는 것이 DI의 핵심.**


### 좋은 코드 만들기 @Lumi

**좋은 코드란 ?**
가독성, 확장성, 유지보수성

**가독성**
- **네이밍을 잘 하자**
	- 의도를 알 수 있게 디테일하게 작성하기
- **Magic Number(String) 제거**
	- 하나의 파일에 묶어서 사용하기 (중요한 건 분리)
- **계산식, 조건문, 로직 다루기**
	- 단순히 삼항 연산자나 조건문만 있으면 어떤 의도로 사용되는지 이해하기 어려울 수 있다.


**분리**
- **Component 분리**
	- 반복되는 컴포넌트만 분리, 모두 다 분리, depth 2칸 이상이면 분리 등등 자신의 **기준**이나 팀 컨벤션에 맞춰 분리.
	- 중요한 건 **기준**에 따라 명확하게 분리하기.

**SOLID 법칙**
- 단일 책임 원칙
- **개방 - 폐쇄 원칙**
	- 확장에는 열려있고 수정에는 닫혀있다.
	- if else를 사용한 로직보다 프로토콜을 만들어두고 필요한 기능이 생기면 클래스를 만들어서 기능을 추가하는 방식 지향.
- 리스코프-치환 원칙
- 인터페이스 분리 법칙
- **의존성 역전 원칙**
	- 구체적인 클래스를 직접 의존하는 것이 아니라, init을 통한 추상화에 의존하기
